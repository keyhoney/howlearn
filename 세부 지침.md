대체로 “그대로 가도 됩니다.” 다만 2026년 기준으로는 몇 군데를 현실적으로 손보는 게 좋아요. (특히 애드센스/SEO는 기술 스택보다 “인덱싱·메타데이터·robots/sitemap·동의(CMP)·ads.txt” 같은 운영 요소가 성패를 갈라서요.)

프레임워크는 Next.js “14 고정”보다는 “최신 안정(stable) 메이저”를 권합니다. 2025년부터 Next.js 15 기준 글/가이드가 이미 꽤 많고(실제로 App Router + AdSense 가이드를 Next.js 15로 다루는 글도 많습니다) , 공식 문서도 App Router 중심으로 계속 업데이트되고 있어요.
즉 “Next.js 14(App Router)” 자체는 충분히 가능하지만, 새 프로젝트라면 굳이 한 메이저 뒤에 머물 이유가 없습니다.

콘텐츠를 MDX로 가는 건 아주 좋습니다. Next.js App Router에서 로컬/원격 MDX 모두 지원하고, Server Components 환경에서도 변환 파이프라인을 공식적으로 안내합니다.
다만 실무적으로는 MDX 렌더링(특히 표/각종 마크다운 확장)이 케이스별로 한 번씩 걸릴 수 있어서(예: 테이블 렌더링 이슈) 글 작성 규칙을 정해두거나 remark/rehype 구성을 초기에 고정하는 게 좋습니다.

스타일(Tailwind), 호스팅(Vercel), 저장소(GitHub)는 무난합니다. 오히려 Vercel은 Next.js 배포/캐싱/ISR 궁합이 좋아서 블로그 운영에 적합해요.

SEO·분석은 “Search Console + GA4”만 적어두면 부족합니다. Next.js App Router에서는 robots.txt, 메타데이터, OG, sitemap 같은 것들이 성능/SEO에 직결되는데, 이건 스택 항목으로 명시해두는 걸 추천합니다. 특히 robots.txt는 App Router에서 공식 파일 컨벤션으로 관리할 수 있습니다. 그리고 Metadata API(정적 metadata / generateMetadata)로 제목·설명·OG를 체계적으로 넣는 게 기본기입니다.

추가로, 애드센스 목적이면 스택 표에 아래 3가지는 “필수 운영 요소”로 같이 들어가는 게 맞습니다.

ads.txt
Next.js는 public 폴더에 둔 정적 파일을 루트로 서빙합니다. 따라서 /ads.txt는 보통 public/ads.txt로 처리하는 게 정석입니다. (공식 문서가 public 폴더의 동작을 명확히 설명합니다.)

서드파티 스크립트 로딩 전략
GA/기타 태그를 성능 나쁘게 붙이면 Core Web Vitals가 흔들릴 수 있어서, Next.js가 제공하는 서드파티 최적화 가이드(@next/third-parties)를 참고해서 로딩을 표준화하는 게 좋습니다.
애드센스도 “전역 레이아웃에 스크립트 1회 로드 + 광고 슬롯 컴포넌트에서 렌더/리프레시” 패턴으로 깔끔하게 가는 편이 운영상 안정적입니다(페이지 전환 시 광고 노출/요청 관리가 포인트).

쿠키/동의(CMP)
심리·교육 콘텐츠는 해외 유입이 생기면 동의 체계가 수익(특히 개인화 광고)과 직결될 수 있습니다. 이건 “기술 스택”이라기보다 “수익 스택”에 가깝지만, 처음부터 포함시키는 게 시행착오가 적습니다.

정리하면, 질문의 표를 “애드센스·SEO 추천 스택”으로 완성형에 가깝게 만들려면 이렇게 바꾸는 걸 권합니다.

프레임워크: Next.js (App Router, 최신 stable 메이저 권장)

콘텐츠: MDX

스타일: Tailwind CSS

호스팅: Vercel

저장소: GitHub

SEO: Metadata API(OG 포함), robots.txt, sitemap.xml, canonical

분석: Search Console, GA4

애드센스 운영: ads.txt, (필요 시) CMP/동의 배너, 서드파티 스크립트 로딩 표준화

원하면, 지금 스택을 기준으로 “블로그에 꼭 필요한 라우트/파일 컨벤션 체크리스트(robots.txt, sitemap, RSS, ads.txt, post 슬러그, 카테고리/태그 아카이브, JSON-LD Article)”를 Next.js App Router 구조로 딱 맞게 적어드릴게요.

응 해줘
9s 동안 생각함

좋아요. Vercel + Next.js(App Router)로 “애드센스/SEO 되는 블로그”를 만들 때, 실제로 필요한 라우트/파일/컨벤션을 한 번에 점검할 수 있는 체크리스트를 Next.js App Router 기준으로 정리해드릴게요. (아래는 “구현 순서”대로 나열했습니다.)

프로젝트 구조 기본

app/ : App Router 루트

app/(site)/ : 사이트 본문 라우트 그룹(선택)

app/blog/[slug]/page.tsx : 글 상세

app/blog/page.tsx : 글 목록(페이지네이션 권장)

app/tags/[tag]/page.tsx : 태그 아카이브(선택)

public/ : 정적 파일(ads.txt, favicon 등)

content/ 또는 posts/ : MDX 원문 저장 폴더(로컬 MDX면 흔히 이렇게 둡니다)

Next.js의 프로젝트 구조/파일 컨벤션은 공식 문서에 정리되어 있습니다.

필수 “정적/메타데이터 파일” 6종 세트

A. robots.txt (필수)

위치: app/robots.txt (정적) 또는 app/robots.ts (동적 생성)

robots.txt 안에 Sitemap URL을 명시하는 패턴이 흔합니다.

B. sitemap.xml (필수)

위치: app/sitemap.xml (정적) 또는 app/sitemap.ts (동적 생성)

섹션별로 sitemap을 쪼갤 수도 있고(예: app/blog/sitemap.xml), generateSitemaps로 다중 사이트맵도 가능합니다.

C. RSS (권장)

Next.js “전용 컨벤션”은 아니고, 보통 Route Handler로 만듭니다.

위치 예: app/rss.xml/route.ts (Content-Type: application/rss+xml)

RSS는 크롤러/구독자/재방문 모두에 도움이 됩니다(애드센스 직접 요소는 아니지만 SEO/재방문 측면에서 꽤 유효).

D. ads.txt (애드센스 필수)

위치: public/ads.txt

배포 후 https://도메인/ads.txt
 로 200 OK가 뜨는지 먼저 확인하세요. Next/Vercel에서 이게 정석이라는 답변이 유지보수자/커뮤니티에서 반복됩니다.

E. favicon / app icons (권장)

App Router는 metadata 아이콘 세팅/정적 아이콘 파일 컨벤션을 지원합니다(구현은 자유지만, 파비콘이 없으면 공유/브랜드/클릭에서 손해).

F. manifest.webmanifest (선택)

PWA까지 갈 생각이 있으면 추가. 아니면 생략 가능.

SEO 핵심: 페이지별 Metadata + canonical + OG

각 page.tsx에서 Metadata API(정적 metadata 또는 generateMetadata)로 title/description/og/twitter를 제대로 채우세요. App Router에서 이게 정석 워크플로우입니다.

canonical은 “중복/쿼리 스트립/페이지네이션”에서 특히 중요합니다(블로그는 목록 페이지, 태그 페이지에서 실수 잦음).

OG 이미지는 글마다 자동 생성하거나(간단한 템플릿) 대표 이미지로 고정해도 됩니다.

구조화 데이터(JSON-LD): Article / BreadcrumbList

블로그 SEO에서 체감이 큰 구간입니다.

글 상세: Article(또는 BlogPosting) JSON-LD

headline, description, datePublished, dateModified, author, image, mainEntityOfPage 등

목록/카테고리: BreadcrumbList(선택)

중요한 건 “과학/심리” 주제에서 E-E-A-T 신호를 주기 좋아서, author를 반드시 넣는 게 유리합니다(저자 페이지와 연결).

인덱싱 안정 장치: 404/리다이렉트/중복 방지

app/not-found.tsx : 404 처리(필수급)

redirects/rewrite 정책(필요 시)과 canonical의 정합성 체크

trailingSlash 유무 통일(한 번 정하면 유지)

성능/크롤링: 렌더링 전략

글 상세/목록은 가능한 SSG/ISR로(“콘텐츠 본문이 HTML로 먼저 나오도록”)

광고 컴포넌트/측정 스크립트만 클라이언트에서 최소로

이미지 next/image 사용(로딩/CLS 방지)

분석: Search Console + GA4 + 이벤트 설계

GA4는 App Router에서 @next/third-parties(또는 next/script)로 표준화하는 게 권장 흐름입니다.

이벤트는 최소 3개만 먼저:

outbound_click (서점 링크 클릭 추적: 리디/교보/예스24/알라딘)

scroll_depth (50%, 90%)

search (사이트 내 검색이 있으면)

애드센스 최적화는 “페이지뷰”보다 “체류/스크롤”이 더 중요하게 작동하는 경우가 많습니다(광고 viewability).

애드센스 삽질 방지 체크(운영)

ads.txt는 “정상적으로 열리는데도” 대시보드가 한동안 못 읽는 경우가 있다고 유지보수자 답변에서도 언급됩니다. 그래서 /ads.txt 200 확인 + 며칠 텀은 감안해야 합니다.

광고 스크립트는 사이트 전체에서 1회 로드, 슬롯은 컴포넌트화(중복 로드/중복 push 방지)

(해외 유입이 생기면) 동의(CMP) 이슈가 수익에 영향을 줄 수 있으니, 최소한 “쿠키/개인정보 고지 페이지”는 초기부터 준비하세요.

블로그에 특히 중요한 “페이지” (신뢰/심사/SEO)

/about (저자 소개: ‘인지심리/뇌과학 기반 수학학습 적용법’ + 전자책 출간 사실)

/privacy (개인정보처리방침)

/terms (이용약관, 선택)

/contact (연락처/문의 폼)

/disclaimer (의학적 진단/치료 아님, 교육 정보 목적 등 짧게)

특히 심리·뇌과학 콘텐츠는 “단정/치료”로 오해받지 않게 고지 페이지가 안전장치가 됩니다.

원하면 다음 단계로, 당신이 쓰려는 글 형태(“학부모 적용 가이드”)에 맞춰서 1) URL 설계(카테고리/태그/시리즈) 2) sitemap 분할 기준(블로그/페이지/태그) 3) Article JSON-LD 필수 필드 템플릿을 바로 써먹을 수 있게 정리해드릴게요.

응

좋습니다. “학부모 적용 가이드(인지심리/뇌과학 기반)”를 애드센스 트래픽형으로 운영한다는 전제에서, (1) URL 설계, (2) sitemap 분할 기준, (3) Article JSON-LD 템플릿(Next.js App Router에서 바로 쓰는 형태)까지 실전형으로 정리합니다.

URL 설계(SEO/확장/운영 난이도 균형)

원칙은 3가지입니다. (a) URL은 짧고 예측 가능, (b) 카테고리/태그는 “사용자 검색 언어”로, (c) 나중에 글이 300개가 되어도 구조가 안 무너지는 것.

권장 베이스 라우트

홈: /

블로그 목록: /blog

글 상세: /blog/[slug]

카테고리: /c/[category]

태그: /t/[tag]

시리즈(선택): /s/[series]

저자 소개: /about

개인정보/쿠키/문의: /privacy, /cookies, /contact, /disclaimer

왜 /blog 아래에 다 넣지 않고 /c, /t를 분리하냐면, 나중에 /blog 자체를 “최신 글/인기 글/검색” 허브로 키우기 편하고, 카테고리/태그는 성격이 다른 아카이브라서 URL 계층을 분리하는 게 운영이 깔끔해집니다.

카테고리 설계(“검색 의도” 중심, 6~10개가 적정)

학부모가 실제로 검색하는 단어들로 카테고리를 잡는 게 중요합니다. “인지부하/작업기억” 같은 학술어를 카테고리로 쓰면 아카이브 페이지 자체의 검색성이 떨어집니다. 대신 아래처럼 “상황/문제” 카테고리를 추천합니다.

예시(시작 8개)

/c/수포자

/c/초등수학

/c/중등수학

/c/수학불안

/c/공부습관

/c/학습동기

/c/오답관리

/c/학부모코칭

태그 설계(“이론/기법”은 태그로)

이론/기법은 태그로 두는 게 자연스럽습니다. 태그는 글이 늘수록 유용해지고, “이론+적용” 교차 탐색이 됩니다.

예시 태그

/t/작업기억

/t/인지부하

/t/회상연습

/t/분산학습

/t/인터리빙

/t/자기효능감

/t/학습된무기력

/t/메타인지

슬러그 규칙(아주 중요)

슬러그는 한글/영문 둘 다 가능하지만 “일관성”이 더 중요합니다.

권장 규칙(실무적으로 안전)

글 슬러그는 영문(또는 영문+숫자)로 통일: /blog/why-kids-give-up-math

화면에는 한글 제목 그대로 노출

내부 데이터에는 slug, title, date, updated, category[], tags[]를 분리

이렇게 하면 URL 인코딩 문제/공유 깨짐/서드파티 도구 호환 이슈가 줄고, 나중에 마이그레이션도 쉬워집니다.

sitemap 분할 기준(규모 커질 때 SEO 안정성)

작게 시작하면 /sitemap.xml 하나로 충분합니다. 다만 블로그는 글이 늘면 사이트맵을 나누는 게 더 안정적이고, 크롤러 효율도 좋아집니다.

권장 분할 구조(처음부터 만들어도 부담 없음)

/sitemap.xml (인덱스 사이트맵)

/sitemaps/pages.xml (정적 페이지: about/privacy/contact 등)

/sitemaps/posts-1.xml (글 상세, 최신부터 N개)

/sitemaps/posts-2.xml (글 상세, 다음 N개)

/sitemaps/categories.xml (카테고리 아카이브)

/sitemaps/tags.xml (태그 아카이브; 태그가 너무 많으면 “상위 태그만” 추천)

/sitemaps/rss.xml (선택: RSS를 사이트맵으로 따로 넣기보단 RSS 자체 유지)

분할 기준(현실적인 컷)

글 상세 사이트맵: 한 파일에 1,000~5,000 URL 정도를 목표로(너무 자주 바뀌는 것/너무 큰 것 모두 비효율)

태그 사이트맵: 태그가 200개를 넘어가면 “전부 넣지 말고” 상위(예: 게시글 3개 이상 연결된 태그)만 포함 추천

카테고리 사이트맵: 카테고리 수가 적으니 전부 포함 OK

robots.txt에 반드시 사이트맵 위치 명시

robots.txt에 Sitemap: https://도메인/sitemap.xml 1줄은 꼭 넣으세요.

그리고 Search Console에 /sitemap.xml 제출.

Next.js App Router 구현 포인트(개념만)

app/sitemap.ts 로 “동적 사이트맵”을 만들 수 있습니다.

규모가 커지면 app/sitemap.ts는 “인덱스 사이트맵”만 만들고, 실제 분할 파일은 app/sitemaps/posts/[id]/sitemap.ts 같은 식으로 라우팅해 생성하는 패턴을 씁니다.

글이 MDX 로컬이면 “빌드 시 읽어와서(SSG) 고정”하거나, ISR로 주기 갱신하는 구조가 운영상 편합니다.

Article JSON-LD 템플릿(복붙해서 바로 적용 가능한 형태)

목표는 “Article/BlogPosting + 저자 + 날짜 + 대표 이미지 + canonical”까지 정확히 넣는 겁니다. 특히 심리/뇌과학 주제는 author/출처/업데이트 날짜가 신뢰 신호로 작동하기 쉽습니다.

아래는 글 상세 페이지(/blog/[slug])에서 <script type="application/ld+json">로 넣는 기본 템플릿입니다. (BlogPosting을 써도 되고 Article을 써도 되는데, 블로그 글이면 BlogPosting이 더 직관적입니다.)

// 예: app/blog/[slug]/page.tsx 내부(또는 별도 컴포넌트)에서 사용
// 주의: 이 객체는 글 데이터(post)로 채워 넣으세요.

const jsonLd = {
  "@context": "https://schema.org",
  "@type": "BlogPosting",
  "mainEntityOfPage": {
    "@type": "WebPage",
    "@id": canonicalUrl, // 예: https://example.com/blog/why-kids-give-up-math
  },
  "headline": post.title,                // 한글 제목 OK
  "description": post.description,       // 1~2문장 요약 (메타 설명과 일치 권장)
  "image": post.ogImageUrl ? [post.ogImageUrl] : undefined, // 가능하면 절대 URL
  "datePublished": post.datePublished,   // ISO 8601: 2026-03-05
  "dateModified": post.dateModified ?? post.datePublished,
  "author": {
    "@type": "Person",
    "name": author.name,                 // 필명/실명
    "url": author.url,                   // /about 절대 URL 권장
  },
  "publisher": {
    "@type": "Organization",
    "name": site.name,                   // 사이트 이름
    "logo": site.logoUrl
      ? { "@type": "ImageObject", "url": site.logoUrl }
      : undefined,
  },
  "inLanguage": "ko-KR",
  "isAccessibleForFree": true,           // 유료벽 없으면 true
  "keywords": post.tags?.join(", "),     // 선택: 태그를 문자열로
};

렌더링 예시(Next.js)

<script
  type="application/ld+json"
  dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
/>

실무 체크 포인트(이거 틀리면 효과가 반감됩니다)

canonicalUrl은 반드시 “절대 URL”이어야 합니다.

datePublished/dateModified는 ISO 8601을 지키세요(YYYY-MM-DD 또는 YYYY-MM-DDTHH:mm:ss+09:00).

ogImageUrl도 가능하면 절대 URL(https://…)로.

author.url은 /about 같은 저자 소개 페이지로 연결. (전자책 출간/전문 분야를 그 페이지에 넣어두면 E-E-A-T 신호가 훨씬 좋아집니다.)

글 목록(/blog)이나 태그/카테고리 아카이브에는 BlogPosting을 넣기보다는, BreadcrumbList 정도가 과하지 않고 안정적입니다.

추가로 넣으면 좋은 BreadcrumbList(선택)

글 상세 페이지에 아래처럼 breadcrumb를 더하면 내부 구조 이해에 도움이 됩니다.

const breadcrumbJsonLd = {
  "@context": "https://schema.org",
  "@type": "BreadcrumbList",
  "itemListElement": [
    { "@type": "ListItem", "position": 1, "name": "홈", "item": site.url },
    { "@type": "ListItem", "position": 2, "name": "블로그", "item": `${site.url}/blog` },
    { "@type": "ListItem", "position": 3, "name": post.title, "item": canonicalUrl },
  ],
};

당신 케이스(애드센스 + 서점 유통 전자책)에 특화된 추천 3가지

서점 링크는 “글 말미에 1회, 고정 문구 + 버튼 1개” 정도로만 두는 게 좋습니다. 너무 상업적으로 보이면 체류가 떨어지고, YMYL 성격 글에서는 신뢰감도 깎일 수 있어요. 대신 GA4 이벤트(outbound_click)로 “어느 글이 서점 클릭을 만드는지”는 꼭 추적하세요.

태그는 무한히 늘리기 쉽습니다. 초기에 규칙을 정하세요.
