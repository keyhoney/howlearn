아래는 당신이 작성한 “Cursor AI 개발 지침서(Next.js App Router + MDX)”를 그대로 기반으로 하되, 지금 목표가 “블로그”가 아니라 “지식 브랜드 사이트(학습과학 교육 사이트)”로 바뀌었기 때문에 반드시 수정/확장해야 하는 부분만 반영해서, 논리 구조에 맞게 다시 엮은 “완성형 개발 계획서”입니다. (원문을 줄이거나 요약하지 않고, ‘바뀐 목표 때문에 달라져야 하는 지점’과 ‘브랜드 사이트로서 추가되어야 하는 지점’만 반영했습니다.)

# Cursor AI 개발 계획서 (학습과학 지식 브랜드 사이트)

이 프로젝트는 Next.js(App Router) + MDX 기반으로 “학습과학(인지심리/뇌과학) 기반 교육 사이트”를 구축한다. 출발 주제는 “왜 아이가 수학을 포기하는가”지만, 향후 전자책을 계속 발행하며 주제를 확장할 계획이므로 사이트 구조는 특정 과목(수학)에 종속되지 않는 “지식 브랜드” 형태로 설계한다. 블로그는 사이트의 일부(SEO 유입 채널)로 유지하되, 사이트의 핵심은 ‘가이드(evergreen)’, ‘개념 사전(용어/이론)’, ‘도구(체크리스트/루틴)’, ‘전자책(Books)’를 포함하는 정보 구조다.

본 지침서는 Cursor AI로 바이브 코딩을 수행할 때 “어떤 디렉토리/라우트/콘텐츠 타입을 어떤 규칙으로 만들고, SEO·애드센스·동의(CMP)·분석까지 무엇을 구현해야 하는지”를 요청 가능한 형태로 명확히 적는다.

## 0. 목표 정의 (블로그 → 지식 브랜드 사이트로 변경되며 반드시 추가되는 요구사항)

기존 “/blog 글 목록 + 글 상세”만 있으면 되는 구조에서, 지식 브랜드 사이트는 최소한 아래 구조가 추가로 필요하다.

1. 사이트 핵심 허브: “가이드(Guide)” 섹션

* 블로그 글과 다르게, “시간순 글”이 아니라 “주제별 핵심 문서(evergreen pillar)”로 구성된다.
* 예: “수학 포기(수포자)의 심리·인지 메커니즘”, “수학 불안(Math Anxiety)”, “작업기억과 수학 문제 해결” 같은 글이 Guide로 편성된다.
* 목적: 신뢰(E-E-A-T) + 내부링크 허브 + 검색 상위의 ‘대표 문서’ 역할.

2. 개념 사전(Concepts / Glossary) 섹션

* 학습과학 용어(작업기억, 인지부하, 회상연습, 분산학습, 자기효능감 등)를 “짧고 표준화된 형식”으로 관리한다.
* 목적: 사이트 전체 용어 통일 + 내부링크(모든 글에서 용어 페이지로 링크) + 장기 SEO 자산.

3. Toolkit(도구/체크리스트/루틴) 섹션

* 학부모 적용형 체크리스트, 루틴 템플릿, 학습 설계 예시 등을 모아 둔다.
* 목적: 체류시간 증가 + 재방문 유도 + 가이드/블로그에서 ‘행동 설계’로 연결.

4. Books(전자책) 섹션

* 리디/교보/예스24/알라딘 판매 링크를 “광고처럼 보이지 않게” 브랜드 페이지로 정리한다.
* 목적: 저자 신호 강화 + 사이트 신뢰도 + 자연스러운 전환(애드센스가 주목표여도 브랜드 사이트에서는 ‘책 존재’가 오히려 신뢰 신호가 됨).

5. 블로그(Blog) 섹션은 유지

* 검색 유입을 만드는 “문제 상황형 글” 중심.
* Guide로 연결되는 ‘지원 콘텐츠(클러스터)’ 역할.

## 1. 프로젝트 기본 설정 (Next.js 버전 정책 반영 및 기존 계획의 수정 지점)

### 1) Next.js 프로젝트 생성

* Next.js는 “최신 안정 메이저(현재 Active LTS)”로 시작한다. 2026년 기준 Next.js 16.x가 Active LTS이고 15.x는 Maintenance LTS이며 14.x는 지원 종료 상태로 표기된다. 따라서 신규 프로젝트는 Next.js 16 기반으로 시작한다. ([nextjs.org][1])
* Cursor AI에게는 `create-next-app`으로 TypeScript, App Router, ESLint, Tailwind를 함께 세팅하도록 지시한다.
* 프로젝트 내부 규칙: app router 기반, server components 기본, 광고/추적만 client component로 최소화.

### 2) Tailwind CSS 설정

* Tailwind 기본 설치 + Typography 플러그인(문서/가이드/블로그 본문 가독성)을 포함한다.
* MDX 본문은 `prose` 기반으로 스타일링하되, 브랜드 사이트이므로 “가이드(긴 글)”에서도 읽기 편한 타이포/간격 규칙을 강제한다.

### 3) MDX 플러그인 설치 및 설정 (기존 계획 유지 + 공식 가이드 기반으로 명시 강화)

* Next.js 공식 가이드는 `@next/mdx` 및 관련 패키지 설치와 `next.config.mjs` 설정, 그리고 App Router에서 `mdx-components.tsx`가 필수임을 명시한다. ([nextjs.org][2])
* 기존 계획의 “mdx-components.tsx 추가”는 유지하되, “App Router에서 없으면 동작하지 않는다”를 필수 조건으로 명확히 적는다. ([nextjs.org][2])

### 4) next.config.mjs

* `createMDX`(withMDX) 적용, `pageExtensions`에 `md`, `mdx` 포함.
* remark/rehype 플러그인은 “초기 고정”이 중요하다. (표/각주/각종 마크다운 확장 때문에 작성 규칙이 흔들리면 유지보수 비용이 커짐)
* 이 단계에서 “frontmatter(메타데이터) 처리 방식”을 결정해야 한다. (아래 2번 섹션에서 구체화)

### 5) mdx-components.tsx

* 전역 MDX 컴포넌트(Heading, Paragraph, Link, Blockquote, Table 등)를 커스터마이징한다.
* 내부 링크는 “개념 사전 페이지로의 링크”를 쉽게 걸 수 있도록 `ConceptLink` 같은 커스텀 컴포넌트를 준비한다(선택).
* 중요한 목표: 사이트 전체 콘텐츠 톤/구조/링크 규칙이 MDX 컴포넌트 레벨에서 강제되게 만든다.

## 2. 콘텐츠 모델/폴더 구조/라우팅 (블로그용 구조에서 브랜드 사이트용 구조로 반드시 확장)

기존 계획은 content/에 blog 포스트만 저장하고 `/blog/[slug]`에서 렌더링하는 구조였다. 지식 브랜드 사이트는 최소 4종의 콘텐츠 타입이 필요하므로 content/ 구조와 라우팅을 확장한다.

### 1) 콘텐츠 타입 정의 (필수 확장)

* blog: 문제 상황형 글(검색 유입, 짧거나 중간 길이)
* guides: 핵심 가이드(evergreen, 긴 문서, 내부링크 허브)
* concepts: 용어/이론 사전(짧고 표준화된 템플릿)
* toolkit: 체크리스트/루틴/템플릿(실전 적용 자료)
* books: 전자책 소개(판매 링크, 서점별 버튼, 책 개요/목차/대상)

### 2) content/ 폴더 구조 (예시)

* `content/blog/*.mdx`
* `content/guides/*.mdx`
* `content/concepts/*.mdx`
* `content/toolkit/*.mdx`
* `content/books/*.mdx` (또는 `content/books/*.json` + 소개 MDX 조합)

“모든 콘텐츠를 MDX로만” 갈 수도 있고, books는 JSON(메타) + MDX(소개 본문)로 분리할 수도 있다. Cursor AI 구현에서는 초기엔 MDX 단일로 통일하고, 확장 시 books만 분리하도록 설계해두는 편이 운영이 편하다.

### 3) 슬러그 규칙(브랜드 사이트 확장 관점에서 수정)

* 원칙: URL은 가능하면 영문 슬러그로 통일한다. (공유/인코딩/도구 호환, 향후 마이그레이션 편의)
* 하지만 categories/tags는 한국어가 SEO에 유리할 수 있으므로, “콘텐츠 슬러그는 영문” + “카테고리/태그는 한글 허용”을 기본으로 한다.
* 예:

  * `/blog/why-kids-give-up-math`
  * `/guides/math-anxiety-mechanism`
  * `/concepts/working-memory`
  * `/toolkit/home-math-routine-10min`

### 4) 라우팅 구조(브랜드 사이트용 확장)

기존:

* `/blog`
* `/blog/[slug]`

확장 후(필수):

* `/` (홈: 브랜드 소개 + 핵심 가이드 허브 + 최신 블로그 + 인기 개념/툴킷)
* `/guides` / `/guides/[slug]`
* `/concepts` / `/concepts/[slug]`
* `/toolkit` / `/toolkit/[slug]`
* `/books` / `/books/[slug]` (또는 `/books` 단일 페이지 + 책 상세는 선택)
* `/blog` / `/blog/[slug]`
* 아카이브:

  * `/c/[category]` (카테고리 아카이브: blog+guides+toolkit 통합 리스트)
  * `/t/[tag]` (태그 아카이브: blog+guides+toolkit 통합 리스트)
  * `/s/[series]` (시리즈/커리큘럼: 선택)

지식 브랜드 사이트에서 아카이브는 “블로그만” 묶지 않고, 가이드/툴킷까지 통합해 “주제 허브”로 작동하게 만드는 편이 SEO/UX에 더 유리하다.

### 5) 프론트매터(메타데이터) 표준 (브랜드 사이트 확장 때문에 반드시 필요)

블로그만 운영할 때는 최소 메타만 있어도 되지만, 브랜드 사이트는 sitemap/JSON-LD/아카이브/내부검색/추천 콘텐츠를 위해 필수 필드가 늘어난다.

모든 콘텐츠 공통 필수 권장 필드:

* `type`: blog | guide | concept | toolkit | book
* `title`
* `description`
* `slug`
* `datePublished` (concept는 선택)
* `dateModified`
* `category` (배열 권장)
* `tags` (배열)
* `grade` (선택: 초등/중등/고등/무관)
* `heroImage` 또는 `ogImage` (선택, 없으면 기본 OG 생성)

books 추가 필드 예:

* `stores`: { ridibooks: url, kyobo: url, yes24: url, aladin: url }
* `isbn`(선택)
* `audience`(학부모/교사/학습코치)
* `toc`(선택: 목차 요약)

### 6) 동적 라우트 구현 방식 (기존 계획 유지하되, 콘텐츠 타입별로 확장)

* 기존처럼 `generateStaticParams`로 슬러그 목록을 만들고 MDX를 import해서 정적 생성(SSG/ISR)을 한다.
* 단, 이제는 `blog`, `guides`, `concepts`, `toolkit`, `books` 각각에 대해:

  * 슬러그 목록 생성 함수
  * 상세 페이지 렌더러
  * 리스트(목록) 페이지
    가 필요하다.

## 3. 사이트 정보 구조(IA)와 내비게이션 (브랜드 사이트로 전환되며 반드시 추가)

블로그는 “목록 → 글”로 끝나도 되지만, 지식 브랜드 사이트는 홈/상단 메뉴/허브 페이지가 SEO와 UX를 좌우한다.

### 1) 글로벌 내비게이션(상단 메뉴)

* Guides
* Concepts
* Toolkit
* Blog
* Books
* About

### 2) 홈(/) 설계(필수)

홈은 단순 최신 글 나열이 아니라 “브랜드 메시지 + 지식 구조”를 보여줘야 한다.

* 1단: 사이트 한 줄 정의(학습과학 기반 학부모 교육 사이트)
* 2단: 대표 가이드 3~6개(핵심 허브)
* 3단: 인기 개념 6~10개(사전형 내부 링크)
* 4단: 최신 블로그 6개(SEO 유입 콘텐츠)
* 5단: Toolkit 추천(체크리스트/루틴)
* 6단: Books 섹션(“전자책 2권 출간” 정도의 사실만 깔끔히)

## 4. SEO 및 애드센스(브랜드 사이트 확장 기준으로 수정/추가)

### 1) Metadata API(기존 계획 유지, 적용 범위 확장)

* 각 라우트별 `title`, `description`, OG, Twitter 카드, canonical을 설정한다.
* blog/guides/concepts/toolkit/books 각각 메타데이터 전략이 다르다.

  * blog: 검색 의도(상황형) + 클릭률(문제 해결형)
  * guides: “대표 문서”로서 더 강한 OG/구조화 데이터
  * concepts: 사전형(정의, 예시, 관련 문서 링크) → OG는 단순하게
  * toolkit: 체크리스트 다운로드/적용 중심 → 체류 유도
  * books: 판매 페이지처럼 보이지 않게 “책 소개/대상/내용” 중심

### 2) robots.txt (기존 계획 유지, Next 공식 컨벤션 명시 강화)

* `app/robots.txt` 또는 `app/robots.ts`로 제공한다. Next.js는 app 루트에 robots 파일을 두는 컨벤션을 명시한다. ([nextjs.org][3])
* robots에 sitemap 위치를 반드시 넣는다.

### 3) sitemap.xml (기존 계획 유지, 브랜드 사이트 확장에 맞게 “분할/인덱스” 권장)

* Next.js는 `app/sitemap.ts`로 동적 생성하는 컨벤션을 제공한다. ([nextjs.org][4])
* 블로그만 있으면 단일 sitemap으로 충분하지만, 브랜드 사이트는 콘텐츠 타입이 늘어나므로 “인덱스 sitemap + 타입별 sitemap” 분할을 고려한다.

  * `/sitemap.xml` (index)
  * `/sitemaps/blog.xml`
  * `/sitemaps/guides.xml`
  * `/sitemaps/concepts.xml`
  * `/sitemaps/toolkit.xml`
  * `/sitemaps/books.xml`
  * `/sitemaps/pages.xml` (about/privacy/contact 등)

### 4) 구조화 데이터(JSON-LD) (기존 계획의 Article JSON-LD를 “콘텐츠 타입별”로 확장)

* blog/guides: `BlogPosting` 또는 `Article`
* concepts: `DefinedTerm` 또는 `Article`(간단하게 `Article`로 통일해도 되지만, 사전형이면 DefinedTerm가 더 적합할 수 있음)
* books: `Book` schema(선택) 또는 `Article` + 판매링크는 본문 링크로 처리
* BreadcrumbList는 전체 콘텐츠 상세 페이지에 공통 적용 가능

핵심은 “저자(author) 정보”를 모든 핵심 문서(guides, blog)에 명시해 E-E-A-T 신호를 강화하는 것이다.

### 5) ads.txt (기존 계획 유지)

* `public/ads.txt`에 게시자 ID를 넣고 배포 후 `https://도메인/ads.txt`가 200으로 열리는지 확인한다.

### 6) CMP/동의(기존 계획 유지, 2026 타임라인을 프로젝트 체크포인트로 명시)

* EEA/UK/CH 트래픽에서 개인화 광고를 위해 “Google 인증 CMP + TCF 문자열 전달”이 사실상 필요하다. Google AdSense는 TCF v2.3 전환 마감(2026-02-28)을 명시하고 있다. ([구글 도움말][5])
* 이 프로젝트 계획서에서는 “CMP를 초기에 완성 구현”까지 강제하기보다, 최소한 다음을 포함한다.

  1. CMP 삽입 지점(레이아웃/스크립트 로딩 위치) 확보
  2. 쿠키 정책(/cookies) + 개인정보처리방침(/privacy) 준비
  3. GA/Ads 스크립트는 ‘동의 상태에 따라 로딩/동작’이 바뀌도록 설계 여지 확보

## 5. 분석(Analytics) 및 서드파티 스크립트 로딩 (브랜드 사이트 기준으로 수정/명시 강화)

### 1) GA4 도입

* Next.js는 서드파티 로딩 최적화를 위해 `@next/third-parties` 사용을 안내한다. ([nextjs.org][6])
* Cursor AI에는 GA4를 `@next/third-parties/google` 또는 `next/script`로 성능 저하 없이 로딩하도록 지시한다. ([nextjs.org][7])

### 2) 이벤트 설계(브랜드 사이트 기준으로 확장)

기존 3개 이벤트(outbound_click / scroll_depth / search)는 유지하되, 브랜드 사이트 특성상 최소 아래까지 확장 여지를 둔다.

* outbound_click: 서점 링크(리디/교보/예스24/알라딘) + 내부 “Books 상세 → 스토어 버튼” 클릭 추적
* scroll_depth: blog와 guides는 각각 다르게(가이드는 25/50/75/90이 더 유의미)
* search: 사이트 내 검색(아래 6번의 검색 기능이 들어갈 경우)
* concept_click: 개념 링크 클릭(내부 링크 성능 측정)
* guide_cta_click: 가이드 내부 “다음 읽을 글/체크리스트” CTA 성능 측정

## 6. 내부 검색(Search) 및 추천/내부링크 (지식 브랜드 사이트에서 새로 추가되는 필수급 기능)

블로그만 운영하면 검색 없이도 버틸 수 있지만, 지식 브랜드 사이트는 “지식 탐색 UX”가 곧 SEO·체류·재방문으로 연결된다. 따라서 최소한의 내부검색 또는 탐색 기능을 계획에 포함시킨다.

### 1) 내부 검색(권장)

* 초기에는 간단한 “클라이언트 검색(정적 인덱스)”로 시작한다.
* 구현 방향:

  * 빌드 시 blog/guides/concepts/toolkit/books의 title/description/tags를 인덱싱
  * `/search` 페이지에서 검색 UI 제공
  * 결과는 콘텐츠 타입별로 그룹핑(guide 먼저, concept, toolkit, blog 순 등)

### 2) 추천 콘텐츠(권장)

* 가이드 상세 페이지 하단: “관련 개념”, “관련 툴킷”, “관련 블로그(클러스터)”를 자동 노출
* 개념 페이지 하단: “이 개념이 포함된 가이드/글 목록” 노출
  이 기능은 “콘텐츠 메타데이터 표준화(2-5)”가 되어 있어야 구현 가능하므로, 계획서에 포함한다.

## 7. 필수 고정 페이지(기존 계획 유지 + 브랜드 사이트용 문서 체계로 정리)

기존에도 about/privacy/contact/disclaimer를 넣었는데, 브랜드 사이트에서는 이 페이지들이 “신뢰 인프라”로 더 중요해진다.

필수:

* `/cookies` (쿠키 정책/CMP 관련 안내)
* `/contact` (문의)
* `/disclaimer` (의학적 진단/치료가 아닌 교육 정보 제공 목적, 전문 상담 권고 문구 포함)

선택:

* `/terms`

## 9. Cursor AI에게 전달할 “실행 지시” 템플릿(블로그 지시를 브랜드 사이트용으로 확장)

아래는 Cursor AI에게 그대로 붙여넣어도 되도록, 작업 단위를 “구현 순서”로 작성한 프롬프트 템플릿이다. (이 템플릿은 기존 블로그 프롬프트를 기반으로, 지식 브랜드 사이트에 필요한 섹션을 추가한 버전이다.)

```text
목표: Next.js(App Router) + Tailwind + MDX 기반 ‘학습과학 지식 브랜드 사이트’를 구현해줘.
사이트는 blog뿐 아니라 guides, concepts(용어사전), toolkit(체크리스트), books(전자책 소개) 섹션을 포함해야 해.

1) 프로젝트 생성
- Next.js 최신 안정(LTS) 기반으로 create-next-app 세팅 (TypeScript, App Router, ESLint, Tailwind 포함).
- Tailwind Typography 플러그인 추가해서 본문에 prose 스타일 적용 가능하게.

2) MDX 구성
- Next 공식 가이드대로 @next/mdx와 관련 의존성 설치.
- next.config.mjs에 createMDX(withMDX) 적용 + pageExtensions에 md/mdx 포함.
- App Router에서 필수인 mdx-components.tsx를 프로젝트 루트에 만들고(useMDXComponents export) 기본 컴포넌트 매핑/스타일을 준비.

3) 콘텐츠 폴더/타입
- content/ 아래에 타입별 폴더 생성:
  - content/blog
  - content/guides
  - content/concepts
  - content/toolkit
  - content/books
- 각 콘텐츠는 MDX로 작성하며 frontmatter 표준을 적용:
  type, title, description, slug, datePublished, dateModified, category[], tags[], grade(optional), ogImage(optional)
  books에는 stores(리디/교보/예스24/알라딘 링크) 필드 추가 가능하게.

4) 라우팅/페이지 구현
- 아래 라우트들을 App Router로 구현:
  / (홈: guides 추천 + 최신 blog + 인기 concepts + toolkit + books 섹션)
  /blog, /blog/[slug]
  /guides, /guides/[slug]
  /concepts, /concepts/[slug]
  /toolkit, /toolkit/[slug]
  /books, /books/[slug] (또는 books는 목록만이라도 우선 구현)
  /c/[category], /t/[tag] 아카이브 (blog+guides+toolkit 통합 리스트)
- 각 [slug] 상세 페이지는 SSG/ISR 기반으로 동작하게 generateStaticParams를 사용.
- 목록 페이지는 페이지네이션(우선은 간단히) 또는 infinite scroll 없이 SSR/SSG로.

5) SEO
- Metadata API로 모든 페이지에 title/description/OG/canonical 설정.
- 콘텐츠 상세 페이지(blog/guides/toolkit)는 JSON-LD(Article/BlogPosting) 삽입. author 정보는 /about과 연결.
- concepts는 사전형 템플릿(정의/핵심요약/예시/관련문서)로 구성하고 메타도 별도 최적화.

6) robots/sitemap
- app/robots.txt 또는 app/robots.ts 구현하고 sitemap 위치 명시.
- app/sitemap.ts 구현. (가능하면 콘텐츠 타입별 분할 sitemap도 고려할 수 있게 설계)

7) 애드센스
- public/ads.txt 생성 가능하게 (실제 publisher ID는 내가 넣을 수 있게 placeholder 처리).
- AdSense 스크립트는 layout에서 1회 로드하도록 구성(중복 로드 방지).
- 광고 슬롯은 컴포넌트화해서 글 본문에 자연스럽게 삽입할 수 있게(예: 첫 H2 이후/중간/하단).

8) CMP/쿠키/고지 페이지
- /privacy, /cookies, /disclaimer, /contact, /about 페이지를 생성.
- CMP는 ‘삽입 가능한 구조’를 먼저 만들고(스크립트 위치/동의 상태에 따른 GA/Ads 로딩 구조), 실제 CMP 벤더 스크립트는 나중에 끼울 수 있게.

9) 분석(GA4)
- @next/third-parties/google 또는 next/script 기반으로 GA4 구성.
- outbound_click(서점 링크), scroll_depth, search 이벤트 훅을 넣을 수 있는 구조를 만들어줘.

10) 최종 검증
- 로컬 빌드(next build) 성공.
- /blog, /guides, /concepts, /toolkit, /books 및 각 상세 페이지 렌더 OK.
- /robots.txt, /sitemap.xml, /ads.txt 경로가 배포 시 정상 노출되도록(public/app 컨벤션 준수).
```

## 10. 이 계획서에서 “원문 대비 변경/확장된 핵심 요약(요약이 아니라 변경점 체크용)”

* 블로그 단일 구조에서 “Guides/Concepts/Toolkit/Books” 섹션이 추가되어 라우팅/콘텐츠 모델/메타데이터/사이트맵 범위가 확장됨.
* Next.js 버전은 2026 기준 Active LTS(16)로 시작하도록 수정됨. ([nextjs.org][1])
* robots/sitemap/MDX/third-parties는 Next 공식 컨벤션/가이드 기준으로 “필수 요건”을 더 명확히 명시함. ([nextjs.org][3])
* CMP/TCF v2.3 전환 마감(2026-02-28)을 “프로젝트 체크포인트”로 명시하고, 초기엔 ‘끼워 넣을 수 있는 구조’부터 만든다는 전략으로 정리함. ([구글 도움말][5])
